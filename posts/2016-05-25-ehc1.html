Intro to Haskell and Functional Programming with Lambda Calculus

Functional Programming

Haskell is a purely functional programming language. It's a general-purpose programming language, meaning, it's not domain specific.
It's functional, computation is viewed as evaluating mathematical functions, avoiding changing-state and mutable data, while managing side effects. It's purely functional, as it excludes destructive modifications, like mutability.

First class values.

So, what is a first class value? First of all, what is a value? A value is anything that can be represented in the computer's memory. A value, for example, can be a number, a string or a date, but not necessarily just those. A programming language defines ways to compute and work with these values, printing them to the screen, or adding them together in case of numbers. So, a “first class value” is simply something that we can pass as an argument, return from a function, or assign to a variable. In a functional language a function can also be assigned to a variable for later computation, or can be passed to another function as a parameter to implement a more generic code.

Functions

The most general definition of a function is a mapping between two sets. A set is just a collection of elements, provided that no element is present multiple times.

Examples of sets would be:

A = {1,2,3}
B = {z,y,x}
C = {0}
D = {} (empty set)

but not these, as elements are not unique in the set.

E = {1,1,2}
F = {z,y,z}

For example, a function f mapping from set A to B, would be:

 f(1) = z
 f(2) = y
 f(3) = x

Or g:

 g(1) = z
 g(2) = z
 g(3) = z

Or h:

 h(1) = z

h is also called a partial function, as it doesn't map each element from A to B, while f and g, which do, are called total functions.

Another way of defining a function is by writing an expression to compute, like so:

 f(x) = x + 1

f denotes the function, x is the argument of the function, while x+1 is the body of the function.
This allows every input to be mapped to a return value computed by the expression.

So, when that gets applied to each member of the set A, it becomes:

 f(1) = 2
 f(2) = 3
 f(3) = 4

Or also:

 f(1) = 1 + 1
 f(2) = 2 + 1
 f(3) = 3 + 1

The two definitions are not distinguishable, as they have the same behaviour.
And this brings us to the main point of this lesson:

Lambda Calculus

From Wikipedia:
Lambda calculus (also written as λ-calculus) is a formal system in mathematical logic for expressing computation based on function abstraction and application using variable binding and substitution.

So, uh, that's a lot. Basically, the lambda calculus is just a formal way of defining a computation. We'll see those abstractions and applications in the following paragraphs, don't worry.

So, let's start defining things, shall we?
Lambda calculus has three lambda terms: expressions, variables and abstractions. Expressions are combinations of variables and abstractions. Variables are just terms for indicating possible inputs. And abstractions are functions, also called lambda. We shall use primarily the Haskell syntax for defining lambda terms.

\x          ->        x
Head     Separator   Body

This is an abstraction. The head of the abstraction is \ (which is shorthand for lambda or ‘lambda’) followed by variable name, and the body is an expression which also might be another abstraction. This means that if we apply an argument to this function, that argument will be rewritten for each x in the body.

(\x -> x) 5
5
(\x -> x * x) 3
3*3

When applying an argument, we also remove the head of the lambda, leaving us with just the internal expression. The argument in the head of the lambda, is called a bound variable, meaning that the value that variable holds, is bound by the argument passed in.

By the way, \x -> x is called the identity function (id), and it's the simplest abstraction, returning the argument given to it.

Alpha Equivalence

The internal name of the variable doesn't actually matter. The following lambdas are specifying the same behaviour.

(\x -> x * x)
(\x' -> x' * x')
(\x'' -> x'' * x'')
(\y -> y * y)
(\z -> z * z)

We can interchange the name of the bound variables, as long as we don't use names already in use, inside the lambda expression, like free variables.

Free variables

These are simply variables which aren't bound by an argument, but occur freely in an expression. In the following examples, z will be used as a free variable.

(\x -> x z)
(\x -> z x)
(\x y -> x y z) (\x -> x)
z
(\x -> x*z+1)

If we take the first expression, and try to use alpha equivalence, to change the name of x to z, we get this.

(\z -> z z)

And this is a vastly different function, because now the parameter will replace both occurrences of z, while, before alpha equivalence, we would have replaced just the first variable x. Compare:

(\x -> x z) 5
5 z
(\z -> z z) 5
5 5

And this means that, when there is a clash of names, there is a need to generate a new identifier. This is usually generated by adding a ' (prime) to the variable name.

(\x -> x z) 5
(\z' -> z' z) 5
5 z

And everything is fine again.

Beta Reduction

And now we reduce. This is the way for the lambda calculus to reduce expressions. The previous examples introduced the reduction implicitly, when we removed an abstraction, by removing the head and separator, and replaced each occurrence of the head variable, with the parameter given.

Another thing: the separator is right associative, while the function application is left associative. This means f 3 2 actually means (f 3) 2, while
\ x -> \y -> x * y means \x -> (\y -> x * y).

Let's look at some examples.

(\x -> x) 3
3

(\x -> x x y) 2
2 2 y

(\x -> x) (\y -> y)
(\y -> y)

(\x -> x z x) (\y -> y)
(\y -> y) z (\y -> y)
z (\y -> y)

And now for some more difficult examples, we shall use also alpha equivalence to avoid name clashes.

(\x -> x y) (\y -> y)
(\y -> y) y 
y

(\x -> \y -> x y) (\y -> y)   -- First we reduce the x

(\y -> (\y -> y) y)           -- Uh-oh, bad case, ABORT!

(\x -> \y -> x y) (\y -> y)   -- So, while reducing, we also need
       [1]    [1]             -- to change the name of y parameter
                              -- at [1], to avoid clashes. 

(\y' -> (\y -> y) y')         -- No more clashes! Let\'s continue 
                              -- with the body of the lambda as
                              -- there are no more arguments outside.

(\y' -> y')                   -- And here we are, with our usual 
                              -- id function again.

When we cannot reduce anymore, we call that expression normal form (actually beta normal form). Each step of reduction also means computing intermediate values. For example, when we have an expression like 3+4 we can reduce this form further by applying the + operator, and getting 7 as the result.

This also introduced…

Multiple Arguments Functions

They don't really exist. Well, maybe. They just are nested lambdas. These three definitions are exactly the same.

(\x y -> x * y)
(\x -> \y -> x * y)
(\x -> (\y -> x * y))

As are these:

(\x y z -> a * x + y * z) -- a is a free variable
(\x -> \y z -> a * x + y * z)
(\x -> \y -> \z -> a * x + y * z)
(\x -> (\y z -> a * x + y * z))
(\x -> (\y -> (\z -> a * x + y * z)))

Multiple parameters are basically functions returning other functions, with one less parameter to fill in.

Combinators

Just a quick note: these are lambda expressions that contain no free variables. They just combine the arguments they were given.

(\x -> x)
(\x y -> y x)
(\x y z -> z x y)
(\x y z -> x y x)
(\x y z -> x*y)
...

Divergence

These are lambda expressions that don't get simpler with beta reduction. They can never be reduced to a normal form.

(\x -> x x) (\y -> y y)   -- substitution of x with (\y -> y y)
((\y -> y y) (\y -> y y))
(\y -> y y) (\y -> y y)   -- With alpha equivalence, we already
                          -- know that we are back at the start.

More examples.

(\b -> \a -> b (\z -> \x -> z x x z) a z) x y
(\a -> x (\z -> \x -> z x x z) a z) y
(x (\z -> \x -> z x x z) y z)
(x (\x -> y x x y) z)
(x y z z y)
(\x y -> y x) ((\y z -> y y z) y x) z -- we shall use the other syntax 
   [1]  [1]                           -- for these multiple parameters.
(\y' -> y' ((\y z -> y y z) y x)) z   -- moved inside the x, renamed
                                      -- y [1] to y\'
(z ((\y z -> y y z) y x))
     [2]     [2]   [3]
(z ((\z -> y y z) x))                 -- y as a bound variable in [2]
                                      -- was replaced with y as a 
                                      -- free variable in [3]
(z ((y y x)))
(z (y y x))
z (y y x)
(\x -> x y) (\z -> z a) (\x y -> y x) x y
((\z -> z a) y) (\x y -> y x) x y
(y a) (\x y -> y x) x y
y a (\x y -> y x) x y -- we cannot reduce anymore, as
                      -- application is left associative
(((y a) (\x y -> y x)) x) y
(\x y -> y x) (\z y -> y z) z
(\y -> y (\z y -> y z)) z -- uh oh, we should have used alpha eq
(\y' -> y' (\z y -> y z)) z
(z (\z y -> y z))         -- this is still fine though, the z at
[1] [2]    [3]            -- [1] is free, [2] bounds [3], no clash
z (\z y -> y z)
